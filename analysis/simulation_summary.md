# Conditional Write Simulation Summary

Statistical summary of YCSB benchmark measurements (June 2025) across
AWS, Azure, and GCP.  Data are separated by **storage tier** and
operation type: **CAS** (compare-and-swap) and **conditional append**.
All latencies were measured at the client and include network round-trip time.

## Data Sources

| Storage Tier | VM Type | CAS | Append |
|--------------|---------|:---:|:------:|
| AWS S3 (Standard) | m5.4xlarge | Yes | No |
| AWS S3 Express One Zone | m5.4xlarge | Yes | Yes |
| Azure Blob Storage | Standard_D16s_v3 | Yes | Yes |
| Azure Premium Block Blob | Standard_D16s_v3 | Yes | Yes |
| GCP Cloud Storage | n2-standard-16 | Yes | No |

Each benchmark ran 5 trials per thread count (1-16 threads), with
concurrent-window trimming applied so that only the time interval where
all JVM instances overlapped is included.

### 9. Data quality notes and caveats

- **GCP** has the smallest dataset (23,635 successful CAS operations total; 2,135 at 1 thread, 1,072 at 16 threads).  GCP's extremely heavy tail (sigma > 0.9 in the lognormal fit) means the sample mean is unstable â€” the apparent *decrease* in mean latency from 1 thread (701 ms) to 16 threads (493 ms) is an artifact of outlier sensitivity, not a real speedup.  The median is stable at ~170 ms across all thread counts.  Use the median or the lognormal parameters, not the mean, for GCP simulation.

- **Azure Blob conditional append** failures are 32x slower than successes (mean 3011 ms vs 95.3 ms).  This suggests failed appends may involve server-side retry or conflict-resolution delays.  AWS failures, by contrast, are only slightly slower than successes.

- **Azure Premium conditional append** failures are 36x slower than successes (mean 2593 ms vs 71.6 ms).  This suggests failed appends may involve server-side retry or conflict-resolution delays.  AWS failures, by contrast, are only slightly slower than successes.

- **S3 (Standard)** was only benchmarked for CAS, not conditional append.  S3 Express One Zone has both.



### 10. Recommended simulation approach

Based on the data:

1. **Model latency as lognormal** with per-cloud, per-tier, per-operation-type parameters.  Use the per-thread fits when simulating specific contention levels, or the aggregate fit for a general model.

2. **Model failure probability** as a function of thread count.  The failure rate is roughly `1 - 1/N` for N concurrent writers under CAS, but the actual rates show provider-specific deviations from this ideal.

3. **Use separate latency distributions for success and failure** paths, as their characteristics differ.

4. **Scale latency with contention**: multiply the base (1-thread) latency by a contention factor derived from the per-thread data.

5. **Account for tail latency**: the p99/p50 ratios indicate that a small fraction of operations will take significantly longer than the median.  The lognormal distribution naturally captures this.

6. **Distinguish storage tiers**: S3 standard and S3 Express One Zone have different latency profiles, as do Azure Blob and Azure Premium.  Do not merge them.
